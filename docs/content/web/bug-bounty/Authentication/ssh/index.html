This guide covers the risks of fallback login methods such as SSH access enabled alongside web login, which may bypass primary authentication controls.

1. What Is the Issue?

When fallback login methods like SSH are enabled without strict controls, attackers can bypass web authentication by directly accessing backend systems.

2. Risks and Impacts

  • Unauthorized access to internal servers and sensitive data.
  • Bypass of web application security mechanisms (e.g., 2FA).
  • Potential for lateral movement and privilege escalation.
  • Increased attack surface and compliance failures.

3. How to Test

  • Scan for open SSH ports on target systems.
  • Attempt to connect using valid or default credentials.
  • Check if SSH access is restricted to authorized users and IPs.
  • Verify if SSH keys are managed securely and regularly rotated.

4. Prevention Strategies

  • Disable SSH access where not required, especially on public-facing systems.
  • Restrict SSH access using firewall rules and IP whitelisting.
  • Use strong authentication methods like key-based SSH login with passphrases.
  • Enforce logging, monitoring, and alerting on SSH logins.
  • Separate admin access from regular user accounts.

5. Example: Restrict SSH Access (Linux Firewall)

# Allow SSH only from trusted IP
sudo ufw allow from 192.168.1.100 to any port 22
sudo ufw deny 22

Summary

  • Fallback login methods like SSH can undermine application security if not tightly controlled.
  • Audit and restrict all alternative login paths rigorously.
  • Implement layered security controls to minimize risks.
